# Req - A Software Prerequisite Checker & Installer


<div align="center">
<img src="doc/img/req-icon.png" alt="Req Logo" width="110px" height="auto">
</div>

<br clear="left"/>

**Req** (aka *JinReq*) is a simple Python based CLI gizmo for checking that
software prerequisites are installed and, optionally, installing them. It is
designed for use in software projects, CI/CD pipelines, makefiles etc.

**Req** uses a simple YAML based format to specify required software components
and the means to check if they are installed and to install them.

## Installing Req
<!-- toc text="" -->

**Req** consists of a single Python file and has only one external package
dependency: **pyyaml**. 

It can be installed from PyPI, but is also trivial to install manually.

```bash
pip install jinreq

# Check that it works
req --help
```

## Getting Started

**Req** uses a simple YAML format specification file to define the requirements
that need to be installed, how to check that they are installed and how to
install them if required.

Here is a simple specification file (`req-demo.yaml`) to cover the
[aspell](http://aspell.net) open source spelling checker.

```yaml
$schema: https://github.com/jin-gizmo/req/releases/latest/download/req.schema.yaml

name: demo
description: Introduction to req (jinreq)

require:
  - name: aspell
    description: Open source spell checker (http://aspell.net)
    if: "test -d doc"
    check: "aspell --version > /dev/null 2>&1"
    install: "brew install aspell"
```

Key points:

*   The `require` key is a list of individual requirements.

*   A *requirement* is anything that can be checked by running a **bash** shell
    script (the `check` key) and installed using a **bash** shell script (the
    `install` key).  This could be the presence of a required application or
    file, or it could be some required configuration arrangement.

*   The optional `if` key is a **bash** script that determines whether the
    requirement applies. A zero exit status indicates the requirement applies.
    In this example, our `aspell` requirement only applies if there is a `doc`
    directory present. A requirement without an `if` script always applies.

*   The `check` key is a **bash** script that determines if the requirement is
    already fulfilled. A `check` condition is satisfied if the script exits with
    zero status.

*   The `install` key is a **bash** script that does whatever is necessary to
    fulfil the requirement. An `install` is considered to have succeeded if the
    script exits with zero status and a subsequent `check` indicates the
    requirement is now fulfilled. (It is possible to bypass this final check.)

*   The `$schema` attribute is optional. This is a published JSON schema for the
    **req** specification file format. **Req** itself doesn't use it, but it is
    handy when editing a specification in an IDE that understands such things
    (e.g.  the Jetbrains suite or VS Code) as it will provide syntax checking
    and auto-completion. For brevity, this will be omitted in subsequent
    examples.

We can use this specification like so:

```bash
# Validate the specification (specification structure, shell script syntax).
req validate req-demo.yaml

# Check the status of our requirements.
req check req-demo.yaml

# Install anything that is missing.
req install req-demo.yaml
```
> [!NOTE]
> Check the [examples](examples/) directory for a few more examples.

## Installation Flow

![Installation Flow](doc/img/install-flow.svg)

As shown above, the installation flow for a requirement is controlled by the
exit status of the `if`, `check` and `install` scripts involved in the process.

Following normal shell conventions, a zero exit status means success or *true*
and, with one exception, a non-zero exit status means failure or *false*.

The exception relates to the `install` step where an exit status of 126 forces
the post-install check to be skipped. See
[Fire and Forget Installs](#fire-and-forget-installs).


## Handling Platform Variations
<!-- toc text="Platform Variations" -->

The example above is fine for macOS with Homebrew installed, but the install
won't work on Linux systems. Since our `check` and `install` scripts are just
**bash**, we can include any amount of logic to handle platform variations.

It can be tedious accommodating the vagaries of the way different platforms
identify themselves. **Req** takes some of the pain out of this by grouping
operating system distros into related families and providing some identifying
environment variables in a uniform format.

The family relationships assumed by **req** are:

| Distro Family | Indicative Members                          |
| ------------- | ------------------------------------------- |
| debian        | debian, ubuntu, mint, raspbian, pop         |
| fedora        | fedora, rhel, centos, rocky, alma, ol, amzn |
| arch          | arch, manjaro, endeavouros                  |
| alpine        | alpine                                      |
| suse          | suse, opensuse                              |

**Req** will attempt to populate the following environment variables, which are
then available to the scripts used in the process of checking and installing
requirements.


|Name|Description|
|-|-|
|REQ_OS|The O/S designation (e.g. `darwin` for macOS and `linux` for Linux). |
|REQ_FAMILY| One of the distro families shown in the table above. This will always be populated with a value. The value of last resort is `unknown`. Use this variable if you need to know whether to install with Homebrew, **dnf** or **apt**, for example.| 
|REQ_ID|The distro ID. This will be populated for well-behaved distros and is usually more specific than `REQ_FAMILY`. Use this variable if you need to differentiate between Debian and Ubuntu, for example.|
|REQ_ID_LIKE|This one is a lottery. Use it at your peril.|
|REQ_VERSION_ID|This one can be useful within the context of a specific distro.|

> [!IMPORTANT]
> The values are always lower case.

> [!NOTE]
> On Linux distros, the contents of `/etc/os-release` are used to set these
> environment variables.
    
The following table shows typical values for common platforms.

| O/S          | REQ_FAMILY | REQ_ID | REQ_ID_LIKE        | REQ_OS | REQ_VERSION_ID |
| ------------ | ---------- | ------ | ------------------ | ------ | -------------: |
| macOS        | darwin     | darwin |                    | darwin |         15.7.4 |
| Debian GNU   | debian     | debian |                    | linux  |             13 |
| Ubuntu       | debian     | ubuntu | debian             | linux  |          24.04 |
| Amazon Linux | fedora     | amzn   | fedora             | linux  |           2023 |
| Fedora       | fedora     | fedora |                    | linux  |             43 |
| Rocky Linux  | fedora     | rocky  | rhel centos fedora | linux  |            9.6 |

> [!NOTE]
> The `req env` command will print the current platform's values for these
> environment variables.

We can now rewrite our **aspell** example like so:

```yaml
name: demo
description: Introduction to req (jinreq)

require:
  - name: aspell
    description: Open source spell checker (http://aspell.net)
    if: "test -d doc"
    check: "aspell --version > /dev/null 2>&1"
    install: |
      case "$REQ_FAMILY"
      in
        darwin)
                brew install aspell
                ;;
        debian)
                sudo apt update
                sudo apt install aspell aspell-en
                ;;
        fedora)
                sudo dnf install aspell aspell-en
                ;;
        arch)
                sudo pacman -S aspell aspell-en
                ;;
        alpine)
                sudo apk add aspell aspell-en
                ;;
        *)
                echo "Try searching your distro package manager index for \"aspell\""
                exit 1
                ;;
      esac
```

## Fire and Forget Installs
<!-- toc text="Fire & Forget Installs" -->

In some circumstances, it is not appropriate to rerun the `check` process
immediately after the `install` process. For example, the `install` process may
just print instructions for the user to follow so running a followup check is
pointless and will always fail. (As Windows is not supported, there should not
be a situation where a reboot is required to make some basic feature work
properly, but that is also a conceivable circumstance in which an immediate
follow-up `check` may be expected to fail.)

If the `install` script exits with a return status of 126, the install is
deemed to have succeeded but the followup `check` process is skipped.

Yes, this can be a crapshoot but is sometimes necessary. Here is our
**aspell** example rewritten to just advise the user how to proceed rather than
actually *install* anything.

```yaml
name: demo
description: Introduction to req (jinreq)

require:
  - name: aspell
    description: Open source spell checker (http://aspell.net)
    if: "test -d doc"
    check: "aspell --version > /dev/null 2>&1"
    install: |
      case "$REQ_FAMILY"
      in
        darwin)
                echo "Try ..."
                echo "    brew install aspell"
                ;;
        debian)
                echo "Try ..."
                echo "    sudo apt update"
                echo "    sudo apt install aspell aspell-en"
                ;;
        fedora)
                echo "Try ..."
                echo "    sudo dnf install aspell aspell-en"
                ;;
        arch)
                echo "Try ..."
                echo "    sudo pacman -S aspell aspell-en"
                ;;
        alpine)
                echo "Try ..."
                echo "    sudo apk add aspell aspell-en"
                ;;
        *)
                echo "Try searching your distro package manager index for \"aspell\""
                exit 1
                ;;
      esac
      exit 126
```

## Shell Helper Functions

**Req** provides a modest library of shell functions that can be used within the
`if`, `check` and `install` **bash** scripts.

> [!NOTE]
> All of the functions provided by **req** have names starting with `req_`.

### Platform Detection Functions

The following functions can be used to check the host platform type. They take
no arguments.

*   `req_is_darwin` / `req_is_macos`
*   `req_is_linux`
*   `req_is_debian`
*   `req_is_fedora`
*   `req_is_arch`
*   `req_is_alpine`
*   `req_is_suse`

For example:

```yaml:
require:
    - name: example
      description: This requirement only applies to Linux systems.
      if: "req_is_linux"
      check: ...
      install: ...
```

### Version Handling Functions

A common requirement is to check / compare version numbers to ensure a target
version of some application is installed. The functions listed below operate on
version numbers in the form `major[.minor[.patch]]`. e.g. `3.4.5`, `3.4`, `3`.
If you need to handle more exotic version syntax, you will need to implement it
yourself in **bash** or some other tool **req** can invoke.

> [!NOTE]
> The functions described consider `3.4.5`, `3.4`, `3` to be *equal*. While this
> may seem odd, it matches a common circumstance in which a tool may report a
> very specific version (e.g. `3.4.5`) but from a requirements point of view you
> just want to make sure you have any version `3.*`.

|Function|Description|
|-|-|
|`req_version_eq`|Compare two versions for equality.|
|`req_version_gt`|Check if the first version is greater than the second.|
|`req_version_gte`|Check if the first version is greater than, or equal to, the second.|
|`req_extract_version`|Extract the first string from stdin that appears to be a version number.|

Examples:

```bash
req_version_eq 3 3.4.5      # True  --> Exit status = 0
req_version_eq 3.4 3.4.5    # True  --> Exit status = 0
req_version_eq 3.5 3.4.5    # False --> Exit status != 0

req_version_gte 3.5 3.4.5   # True  --> Exit status = 0

req_version_gt 3 3.4.5      # False --> Exit status != 0

# Check if our version of bash is v4 or above (looking at you Apple!)
req_version_gte "$(bash --version | req_extract_version)" 4 && echo "We have v4+"
```

### File System Functions

File system functions assist with checking the existence of objects in the local
file system.

|Function|Description|
|-|-|
|`req_has_command`|Check if the specified command is available on PATH. It will not detect shell functions or aliases.|
|`req_has_dir`|Check if the specified directory exists.|
|`req_has_file`|Check if the specified file exists and is a regular file.|

Examples:

```bash
req_has_command aspell      # Check if aspell is installed and on the PATH
req_has_dir /usr/local/bin  # Check if directory /usr/local/bin exists
```

## Optional Requirements

Some requirements may be nice to have but are not essential. These can be marked
with the `optional` key with a boolean value as part of the requirement
specification (default is `false`). By default, **req** will ignore optional
requirements when performing checks and installs unless the `--optional` flag
is used.

As an example, this is what an optional installation of the Esperanto dictionary
for **aspell** may look like as an extension of the earlier examples. (And who
amongst us has never needed to do that!)

```yaml
name: demo
description: Introduction to req (jinreq)

require:
  - name: aspell
    description: Open source spell checker (http://aspell.net)
    if: "req_has_dir doc"
    check: "req_has_command aspell"
    install: ...

  - name: aspell-esperanto
    description: Esperanto dictionary for aspell.
    optional: true
    if: "req_has_command aspell"
    check: |
      data_dir="$(aspell config data-dir)"
      req_has_file "${data_dir}/esperanto.alias"
    install: |
      dict="aspell6-eo-2.1.20000225a-2"
      tmp=$(mktemp -d)
      trap '/bin/rm -rf "$tmp"' EXIT
      cd "$tmp"
      curl -o "${dict}.tar.bz2" "https://ftp.gnu.org/gnu/aspell/dict/eo/${dict}.tar.bz2"
      tar xf "${dict}.tar.bz2"
      cd "$dict"
      ./configure
      make
      make install
```

This would be invoked thus:

```bash
req check --optional req-demo.yaml
req install --optional req-demo.yaml
```

## Generating Documentation from Requirements
<!-- toc text="Generating Doco" -->

The **req** `doc` subcommand auto-generates basic documentation from a
requirements specification file. This can be handy when generating documentation
as part of a CI/CD process.

By default, a very basic Markdown table structure is generated. A custom
template can be supplied using the `template` key in a requirements
specification file. This is the default `template` key:

```yaml
name: demo
description: Introduction to req (jinreq)

template:
  prologue: |
    ### ${name} - ${description}

    | # | Name | Description |
    |:-:|------|-------------|

  requirement: |
    | ${index} | ${name} | ${description} |

  epilogue: "\n"
    
```

The `prologue` is printed first. The following variables can be injected using
`$` style variable substitution `$var` or `${var}`. `$$` is replaced with a
single `$`.

*   `name`: The `name` value from the requirements specification file.
*   `description`: The `description` value from the requirements specification
    file.
*   `source`: The requirements specification file name.

The `requirement` element is printed once for each requirement. The following
variable substitutions are available:

*   `name`: The requirement name.
*   `description`: The requirement description.
*   `optional`: `true` or `false`, depending on whether the requirement is
    optional or not.
*   `index`: The requirement number (starting from 1).

The `epilogue` is printed last. The same variable substitutions as for
`prologue` are supported.

As an example, this is a simple numbered list version:

```yaml
template:
  prologue: |
    ${source} -- ${name} -- ${description}
    
    Requirements
    ------------

  requirement: |
    ${index}.  ${name} -- ${description}

  epilogue: "\n"
```

> [!NOTE]
> Yes, this is a very basic capability and could have been much better using
> something like Jinja ... at the cost of requiring a large and complex package.

## Debugging

The `check` and `install` subcommands have a `-x` / `--trace` option that
enables `-x` mode when running the **bash** scripts for the `if`, `check` and
`install` scripts. This can be handy to see the commands as executed.

By default, the `check` and `install` subcommands will suppress stderr from the
scripts when the script exits successfully and tracing is not enabled. To force
stderr to be emitted, use the `--verbose` option.

Prerequisite checker and installer for software projects.

  {check,doc,env,install,validate}

Check that specified requirements are satisfied.

positional arguments:

## Requirements Specification File Reference
<!--toc text="Requirements Spec Reference" -->

The requirement specification file looks like the following:

```yaml
$schema: https://github.com/jin-gizmo/req/releases/latest/download/req.schema.yaml

name: Name of the requirements set
description: Description of the requirements set

# Optional template for auto generated doco
template:
  prologue: Template text prior to list of requirements
  requirement: Template text for each requirement.
  epilogue: Template text after list of requirements.

# List of requirements to check/install
require:
  - name: Name of requirement 1
    description: Description of requirement 1
    optional: false
    if: Bash script to determine of requirement applies here
    check: Bash script to check if requirement is fulfilled
    install: Bash script to fulfil the requirement

  - name: ...
```

|Name|Type|Required|Description|
|-|-|-|-|
|$schema|String|No|This is a published JSON schema for the **req** specification file format. **Req** itself doesn't use it, but it is handy when editing a specification in an IDE that understands such things (e.g. the Jetbrains suite or VS Code) as it will provide syntax checking and auto-completion.|
|description|String|Well...|A description for the requirements set. **Req** itself doesn't require this but common sense does.|
|name|String|Yes|A short name for the requirements set.|
|require|List[Map]|Yes|A list of individual requirements.|
|template|Map|No|A template for [generating documentation](#generating-documentation-from-requirements) from a requirements specification file. A simple markdown table format is provided as a default.|

Each element of the `require` list is a map (object) with the following
structure:

|Name|Type|Required|Description|
|-|-|-|-|
|check|String|Yes|A **bash** script that determines if the requirement is already fulfilled. A check condition is satisfied if the shell script exits with zero status.|
|description|String|Well...|A description for the requirement. **Req** itself doesn't require this but common sense does.|
|if|String|No|A **bash** script that determines if the requirement applies. If the script exits with zero status, the requirement applies. If not specified, it is assumed that the requirement does apply. See [Installation Flow](#installation-flow) for details.|
|install|String|No|A bash script that does whatever is necessary to fulfil the requirement. An install is considered to have succeeded if the shell script exits with zero status.|
|name|String|Yes|Name of the requirement. Typically, this just identifies the thing that needs to be installed.|
|optional|Boolean|No|Whether or not the requirement is optional. See [Optional Requirements](#optional-requirements).|

## Working with the Repo

To work with the repo:

```bash
# Clone the repo
git clone git@github.com:jin-gizmo/req.git
cd req

# Get help
make

# Initialise the virtual environment.
make init
source venv/bin/activate
```

> [!NOTE]
> The makefile uses [MakeHelp](https://github.com/jin-gizmo/makehelp)
> to auto-generate help.
